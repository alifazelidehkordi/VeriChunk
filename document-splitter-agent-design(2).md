# سند طراحی: ایجنت تقسیم‌کننده‌ی مفهومی اسناد (Document Splitter Agent)

## ۱. هدف

ایجنتی که:
- به‌عنوان **MCP Server** توسط CLIهای مختلف AI (Claude Code، Cursor، Windsurf، Cline و...) قابل شناسایی و استفاده باشد.
- به‌صورت **مستقل (standalone CLI)** هم قابل اجرا باشد.
- فایل ورودی (PDF یا DOCX) را به بخش‌های ۵ تا ۱۰ صفحه‌ای، بر اساس **مرز مفهومی** (نه مرز فیزیکی صفحه) تقسیم کند.
- هیچ بخشی از محتوای اصلی (متن، جدول، تصویر) گم، تکراری یا ناقص نشود — با یک مرحله‌ی Verification تضمین‌شده.

---

## ۲. معماری کلی

```
Input (pdf/docx)
   │
   ▼
[1] Format Detector
   │
   ▼
[2] Parser Agent  ──► IR مشترک (JSON ساختاریافته)
   │
   ▼
[3] Structure Analyzer  ──► درخت heading + word count تجمعی
   │
   ▼
[4] Boundary Detector  ──► نقاط قطع مفهومی
   │
   ▼
[5] Writer Agent  ──► فایل‌های .md + تصاویر استخراج‌شده
   │
   ▼
[6] Verifier Agent  ──► تضمین صحت و کامل بودن خروجی
   │
   ▼
[7] Content Analyzer  ──► خوانش عمیق هر چانک + بازبینی انسجام مفهومی
   │
   ▼
[8] Index Generator  ──► دو فایل ایندکس مطالعاتی (فارسی + انگلیسی)
   │
   ▼
output/ (chunks + manifest + گزارش verification + گزارش انسجام + دو فایل ایندکس)
```

رابط بیرونی (MCP + CLI مستقل) هر دو به همین Orchestrator وصل می‌شوند؛ فقط نقطه‌ی ورودی فرق دارد.

---

## ۳. نمایش میانی مشترک (IR)

هر دو فرمت ورودی (PDF, DOCX) قبل از پردازش به این ساختار یکسان تبدیل می‌شوند تا بقیه‌ی پایپ‌لاین فرمت‌مستقل بماند:

```json
{
  "elements": [
    { "id": "el-001", "type": "heading", "level": 1, "text": "..." },
    { "id": "el-002", "type": "paragraph", "text": "..." },
    { "id": "el-003", "type": "table", "rows": [["...", "..."]] },
    { "id": "el-004", "type": "list", "items": ["...", "..."] },
    { "id": "el-005", "type": "image", "ref": "img-01.png", "caption": "..." }
  ],
  "meta": {
    "source_file": "book.pdf",
    "total_word_count": 48213,
    "estimated_total_pages": 210
  }
}
```

هر element یک `id` یکتا دارد — مبنای چک نهایی Verifier.

### تفاوت استخراج بر اساس فرمت

| جنبه | PDF | DOCX |
|---|---|---|
| ابزار پیشنهادی | `pymupdf4llm` (اصلی) + `OpenDataLoader PDF` (تأیید/دقت موقعیت) | `python-docx` / `mammoth` |
| Heading | تشخیص خودکار از فونت‌سایز یا از فهرست مطالب (TOC) سند | مستقیم از استایل Heading 1/2/3 خوانده می‌شود |
| جدول | استخراج اولیه با pymupdf4llm + تأیید ساختار با OpenDataLoader (دقت جدول بالاتر) | ساختاریافته در XML، مستقیم قابل استخراج |
| هدر/فوتر تکراری | به‌صورت خودکار توسط pymupdf4llm فیلتر می‌شود | جدا از بدنه است |
| OCR | غیرفعال (خارج از scope) — فقط اسناد با متن embedded پشتیبانی می‌شوند | موضوعیت ندارد |
| واحد اندازه‌گیری طول | صفحه‌ی فیزیکی دقیق + bounding box هر عنصر (از OpenDataLoader) | ندارد → تخمین با تعداد کلمه (هر صفحه ≈ ۳۰۰-۵۰۰ کلمه، قابل تنظیم) |

## ۴. جزئیات ساب‌ایجنت‌ها

### [1] Format Detector
ورودی: مسیر فایل → خروجی: نوع فایل + انتخاب Parser مناسب.

### [2] Parser Agent

**برای DOCX:** ساده — `python-docx`/`mammoth` مستقیماً ساختار (heading، جدول، تصویر) را با دقت کامل می‌دهند، چون DOCX ذاتاً ساختاریافته است.

**برای PDF — استراتژی دوموتوره:**

| موتور | نقش |
|---|---|
| **pymupdf4llm** (اصلی) | استخراج سریع و بدون GPU؛ تبدیل مستقیم به Markdown/JSON با `page_chunks=True` (متادیتای هر صفحه) و `write_images=True` (استخراج تصاویر). **OCR کاملاً غیرفعال است** (`use_ocr=False`) — این نسخه فقط اسناد با متن embedded را پشتیبانی می‌کند. |
| **OpenDataLoader PDF** (تکمیلی) | یک پارسر deterministic و rule-based (بدون LLM، بدون GPU) که خروجی JSON با **bounding box دقیق برای هر عنصر** و الگوریتم XY-Cut++ برای ترتیب خواندن در لایه‌های پیچیده (چندستونی، جدول تودرتو) می‌دهد. دقت تشخیص جدولش در بنچمارک رسمی بالاتر از اکثر ابزارهای مشابه گزارش شده. (این ابزار هم در همان حالت local/deterministic استفاده می‌شود، بدون فعال کردن حالت hybrid/OCR.) |

**رفتار در برابر صفحات بدون متن قابل‌استخراج (تصمیم قطعی):**
چون OCR کلاً خارج از scope است، اگر Parser Agent به صفحه‌ای برسد که متن قابل‌انتخاب ندارد (یعنی برای خواندنش OCR لازم بود):
- آن صفحه **رد می‌شود** (skip) — نه پردازش می‌شود، نه با OCR جبران می‌شود.
- در `verification-report.json` با وضوح کامل علامت‌گذاری می‌شود: کدام صفحه، چرا رد شده («نیاز به OCR — خارج از scope فعلی»).
- بقیه‌ی سند به همان روال عادی پردازش می‌شود.
- **نکته‌ی مهم برای Verifier:** چون این حالت یک استثنای شناخته‌شده و از‌پیش‌تعریف‌شده است (نه یک باگ)، چک Coverage باید این صفحات را از محاسبه‌ی «افت محتوا» جدا کند — یعنی گزارش باید صریح بگوید «X صفحه به دلیل نیاز به OCR رد شد» به‌جای اینکه به‌اشتباه به‌عنوان خطای verification نمایش داده شود.

**مرحله‌ی Reconciliation (تطبیق دو خروجی):**
1. متن پایه‌ی هر element از خروجی pymupdf4llm گرفته می‌شود (چون سریع‌تر است و برای متن embedded کاملاً کافی است).
2. موقعیت دقیق صفحه و مختصات هر element — به‌ویژه برای مواردی که یک پاراگراف یا جدول بین دو صفحه ادامه پیدا می‌کند — از bounding boxهای OpenDataLoader گرفته می‌شود. این دقیقاً همان چیزی است که برای محاسبه‌ی صحیح «صفحات ۱ تا ۸» در manifest و ایندکس مطالعاتی لازم داریم.
3. اگر ساختار یک جدول بین دو موتور ناهم‌خوان بود (تعداد ردیف/ستون فرق داشت)، نسخه‌ی OpenDataLoader (دقت جدول بالاتر) در اولویت قرار می‌گیرد و این تصمیم در `verification-report.json` ثبت می‌شود (شفاف، نه بی‌صدا).
4. برای تصاویر: pymupdf4llm فایل را استخراج/ذخیره می‌کند، و bounding box دقیق‌تر OpenDataLoader برای تعیین مرز دقیق ناحیه‌ی تصویر (crop تمیزتر) استفاده می‌شود.

**نکته‌ی مهم:** چون هر دو موتور deterministic هستند (نه LLM-based)، این لایه‌ی دوم هیچ ریسکی برای تغییر محتوا ندارد — فقط دقت موقعیت‌یابی و ساختار را بالا می‌برد. کاملاً هم‌راستا با اصل «متن اصلی هرگز توسط مدل بازنویسی نمی‌شود».

**ملاحظه‌ی عملی برای Deployment:** OpenDataLoader یک CLI جاواست (با wrapper پایتون/Node)، یعنی محیط اجرا به JVM نیاز دارد — این یک وابستگی اضافه است که باید در نصب/کانتینر لحاظ شود.

**سایر وظایف مشترک Parser Agent:**
- تبدیل خروجی به IR بالا.
- فیلتر کردن هدر/فوتر تکراری (خودکار در pymupdf4llm).
- **استخراج تصاویر**: هر تصویر به یک فایل جدا (`images/img-XX.png`) ذخیره می‌شود و در IR فقط رفرنس آن نگه داشته می‌شود (تصمیم قطعی‌شده).

### [3] Structure Analyzer
ساخت درخت سلسله‌مراتبی heading (H1 > H2 > H3) و محاسبه‌ی `cumulative_word_count` روی ترتیب elementها، برای اینکه Boundary Detector بداند هر نقطه از سند معادل چند صفحه است.

### [4] Boundary Detector — بازطراحی‌شده بر پایه‌ی درک مفهومی واقعی

این دقیقاً همون بخشی‌ست که باید از دام «اسکریپت heuristic که فقط heading رو می‌شماره» خارج بشه. طراحی قبلی (heading = مرز) دقیقاً همون مشکلیه که خودت تجربه کردی: heading همیشه با مرز واقعیِ یک مفهوم یکی نیست — گاهی یک heading کوچیک (مثلاً «مثال ۲») وسط یک بحث بزرگ‌تره و نباید ازش جدا بشه، و گاهی بدون هیچ heading جدیدی، موضوع کاملاً عوض می‌شه. برای همین، تصمیم‌گیری رو به دو لایه‌ی جدا می‌شکنیم:

**لایه‌ی ۱ — تولید نقاط قطع «امن» (کاملاً deterministic، بدون LLM):**
روی IR، تمام نقاطی که از نظر ساختاری قطع در آن‌ها بی‌خطر است (پایان پاراگراف، پایان جدول، پایان لیست، شروع/پایان heading) به‌عنوان «کاندیدهای مجاز» علامت‌گذاری می‌شوند. این لایه هیچ تصمیمی درباره‌ی مفهوم نمی‌گیرد — فقط می‌گوید «این نقاط از نظر ساختاری قطع کردن در آن‌ها امن است» (هیچ‌وقت وسط جدول/لیست/جمله نیست). این همون تضمین «بدون افت محتوا» رو حفظ می‌کنه.

**لایه‌ی ۲ — تصمیم‌گیری واقعی، توسط LLM که محتوا رو می‌خونه (نه heading رو می‌شماره):**
این لایه، مغز اصلی ماجراست:
1. از نقطه‌ی فعلی سند، یک بازه‌ی سخاوتمندانه از متن (مثلاً تا ۱.۵ برابر `max_pages`، یعنی حدود ۱۵ صفحه، به‌عنوان بافر) به همراه تمام کاندیدهای امنِ لایه‌ی ۱ در همون بازه، به‌صورت کامل به یک مدل داده می‌شود.
2. از مدل خواسته می‌شود که واقعاً محتوا رو بخونه و بگه: «مفهوم/بحثی که الان در حال توضیح دادنه، کجا واقعاً کامل می‌شه و یک موضوع مستقل جدید از کجا شروع می‌شه؟» — نه «heading بعدی کجاست؟».
3. مدل فقط مجاز است از بین همون کاندیدهای امنِ لایه‌ی ۱ یکی را به‌عنوان نقطه‌ی قطع انتخاب کند (نه هر جای دلخواه) — یعنی می‌تواند تصمیم مفهومی بگیرد، ولی هرگز نمی‌تواند وسط جدول/جمله را انتخاب کند و هرگز متن را تغییر نمی‌دهد؛ کد دقیقاً از همان اندیس، متن اصلی را می‌بُرد.
4. اگر مدل تشخیص بدهد که مفهوم فعلی حتی تا انتهای بازه‌ی ۱۵ صفحه‌ای هم کامل نشده، بازه را بزرگ‌تر می‌کنیم و دوباره می‌پرسیم — این همان «قانون طلایی» قبلی است، ولی حالا به‌جای «heading جدید نیامد پس ادامه بده»، معیار واقعی «مفهوم» است.
5. مدل باید یک توضیح کوتاه هم بدهد که چرا این نقطه را انتخاب کرده (مثلاً «چون مثال‌های ۱ تا ۳ همگی بخشی از یک استدلال واحدند»)؛ این توضیح در `manifest.json` ذخیره می‌شود تا تصمیم‌ها قابل ممیزی باشند، نه یک جعبه‌ی سیاه.

**لایه‌ی ۳ — بازبینی انسجام نهایی:** این بخش به یک ساب‌ایجنت مستقل منتقل شده — ببین [7] Content Analyzer در ادامه، چون این بازبینی به محتوای کامل چانک‌های نهایی نیاز داره، نه فقط تصمیم‌های لحظه‌ای برش.

**نکته‌ی صریح درباره‌ی هزینه:** این طراحی به‌جای یک اسکریپت سریع و ارزان، چند فراخوانی LLM به‌ازای هر چانک (و یک پاس نهایی) نیاز دارد — یعنی کندتر و گران‌تر از heuristic خالصه. این دقیقاً همون trade-off ایه که برای «تقسیم‌بندی واقعاً مفهومی» لازمه؛ اگه بخوایم ارزون و سریع باشه، برمی‌گردیم به همون heuristic ناامیدکننده‌ای که قبلاً تجربه کردی.

**قوانین ثابت (بدون تغییر):**
- جدول همیشه واحد اتمی است؛ هرگز بین دو چانک تقسیم نمی‌شود.
- هیچ‌وقت وسط پاراگراف/جمله قطع نمی‌شود.
- بازه‌ی ۵ تا ۱۰ صفحه یک هدف نرم است؛ اگر مفهوم ادامه داشته باشد، چانک بزرگ‌تر می‌شود.

### [5] Writer Agent
هر چانک را به یک فایل Markdown می‌نویسد، شامل متن، جدول‌ها (به‌صورت جدول Markdown) و لینک به تصاویر مربوطه. هر فایل یک هدر متادیتا دارد:

```markdown
<!-- chunk: 03/18 | source: book.pdf | pages: ~24-31 -->
<!-- continues-from: chunk-02.md | continues-to: chunk-04.md -->

## عنوان بخش
متن... جدول... ![توضیح تصویر](images/img-05.png)
```

### [6] Verifier Agent
سه چک روی خروجی نهایی:
- **Coverage**: هر `id` از IR اصلی دقیقاً یک‌بار در خروجی‌ها ظاهر شده باشد (نه صفر، نه بیشتر از یک) — به‌جز آن دسته از صفحاتی که به‌دلیل نیاز به OCR از قبل به‌صورت شفاف در گزارش «رد شده» علامت‌گذاری شده‌اند (این‌ها استثنای شناخته‌شده‌اند، نه افت محتوای واقعی).
- **Word Count**: مجموع کلمات همه‌ی چانک‌ها با `total_word_count` سند اصلی برابر باشد (± تلورانس فاصله‌ها).
- **Table/Image Integrity**: تعداد ردیف/ستون هر جدول و رفرنس هر تصویر با نسخه‌ی اصلی مطابقت دارد.

اگر هرکدام fail شود، عملیات متوقف می‌شود و گزارش دقیق (کدام element، کجا) تولید می‌شود — به‌جای تحویل بی‌صدای خروجی ناقص.

### [7] Content Analyzer — خوانش عمیق و تحلیل مفهومی هر چانک نهایی

این ساب‌ایجنت دقیقاً پاسخ به نگرانی اصلی توئه: نمی‌ذاره توضیحات ایندکس، خلاصه‌های قالبی/عمومی باشن. فقط بعد از Verifier اجرا می‌شه (روی چانک‌های نهایی و تأییدشده).

**برای هر چانک، مدل واقعاً کل محتوای همون چانک رو می‌خونه** (نه فقط heading یا چند خط اول) و دو خروجی تولید می‌کنه:

1. **توصیف مفهومی دقیق (نه قالبی):**
   - نه یک جمله‌ی ژنریک مثل «این بخش درباره‌ی مبحث X است»، بلکه توصیفی که واقعاً نشون بده داخل اون چانک چه استدلال/مفهوم/مثال‌هایی مطرح شده.
   - اگر یک چانک چند زیرموضوع مرتبط رو پوشش می‌ده (چون طبق طراحی، چند heading کوچیک می‌تونن داخل یک چانک مفهومی جمع شده باشن)، همه‌ی این زیرموضوع‌ها در توصیف ذکر می‌شن، نه فقط heading اول.
   - خروجی به‌صورت ساختاریافته (JSON) و هم‌زمان به هر دو زبان (`topic_fa`, `topic_en`) تولید می‌شه.
2. **پرچم انسجام (Coherence Flag):** مدل درباره‌ی خود چانک قضاوت می‌کنه:
   - آیا این چانک واقعاً یک واحد مفهومی منسجم است؟
   - آیا نسبت به چانک قبل/بعد، این برش منطقی به‌نظر می‌رسد یا جایی احساس می‌شود بحث به‌طرز عجیبی قطع شده؟
   - نتیجه (`confident` / `needs_review` + دلیل کوتاه) در `semantic-review-report.json` ذخیره می‌شود.

**چرا این جدا از Boundary Detector است:** Boundary Detector فقط تصمیم می‌گیرد «کجا برش بخورد»؛ Content Analyzer محتوای نهایی و کامل هر چانک را می‌خواند تا هم توصیف درست تولید کند و هم به‌عنوان یک لایه‌ی QA مستقل، تصمیم‌های Boundary Detector را دوباره از زاویه‌ی دیگری چک کند — دو مدل/دو نگاه، نه یک تصمیم یک‌باره.

**خروجی این مرحله مستقیماً به [8] Index Generator تغذیه می‌شود.**

### [8] Index Generator — ایندکس مطالعاتی

فقط بعد از موفقیت Verifier اجرا می‌شود (روی خروجی تأییدشده کار می‌کند، نه روی چیزی که ممکنه ناقص باشه).

**ورودی:** `manifest.json` (لیست چانک‌ها، محدوده‌ی صفحات، تعداد کلمات هر چانک) + خروجی [7] Content Analyzer (توصیف مفهومی دقیق هر چانک به هر دو زبان).

**مراحل:**

1. **محاسبه‌ی زمان مطالعه‌ی هر بخش**:
   `estimated_minutes = word_count / reading_speed_wpm`
   با `reading_speed_wpm` پیش‌فرض ۲۰۰ کلمه در دقیقه (سرعت مطالعه‌ی دقیق/مطالعاتی، نه مطالعه‌ی سریع) — قابل تنظیم.
2. **محاسبه‌ی زمان کل** = مجموع زمان تمام بخش‌ها.
3. **عنوان و موضوع هر بخش**:
   - عنوان از heading همان چانک گرفته می‌شود (مستقیم از IR، بدون تغییر).
   - موضوع مستقیماً از خروجی Content Analyzer گرفته می‌شود — یعنی توصیفی که واقعاً از خوانش کامل محتوای چانک به‌دست آمده، نه یک برچسب سطحی تولیدشده در همین مرحله. Index Generator خودش هیچ خوانش/تحلیل جدیدی روی متن انجام نمی‌دهد؛ فقط داده‌ی از‌پیش‌تحلیل‌شده را رندر می‌کند.
4. **گروه‌بندی اختیاری در قالب فصل‌های مفهومی**:
   - روش پیش‌فرض (بدون نیاز به LLM، قطعی و قابل‌اعتماد): اگر سند اصلی heading سطح‌بالا (H1) داشته باشد که چند چانک زیرمجموعه‌ی آن قرار گرفته‌اند، همان ساختار به‌عنوان "فصل" در نظر گرفته می‌شود.
   - روش کمکی (اختیاری، با فلگ جدا `use_llm_chapter_grouping`): وقتی سند چنین ساختاری ندارد، یک فراخوانی LLM عناوین/موضوعات بخش‌ها را می‌بیند و بخش‌های مرتبط را زیر یک فصل مشترک گروه‌بندی می‌کند. این گروه‌بندی فقط روی متادیتای نمایشی اثر می‌گذارد، نه روی خود فایل‌های چانک.
   - اگر نه ساختار heading و نه گروه‌بندی معنایی به نتیجه‌ی روشنی نرسند، فصل‌بندی حذف می‌شود و فقط لیست تخت بخش‌ها نمایش داده می‌شود (چون طبق گفته‌ی خودت فصل‌بندی الزامی نیست).
5. **رندر خروجی در دو فایل کاملاً جدا**: `study-index-fa.md` و `study-index-en.md`.

**ساختار هر فایل ایندکس:**

```markdown
# ایندکس مطالعاتی — <نام سند>

## نمای کلی
- تعداد کل بخش‌ها (جلسات مطالعه): 32
- تعداد فصل‌های مفهومی: 5   (در صورت وجود گروه‌بندی)
- زمان تقریبی کل مطالعه: 14 ساعت و 20 دقیقه

## فهرست فصل‌ها و بخش‌ها

### فصل ۱: <عنوان فصل مفهومی> (در صورت وجود گروه‌بندی)

| # | عنوان بخش | موضوع | صفحات منبع اصلی | زمان تقریبی مطالعه |
|---|---|---|---|---|
| 1 | ... | توضیح یک‌خطی موضوع | 1–8 | ~25 دقیقه |
| 2 | ... | ... | 8–15 | ~30 دقیقه |

### فصل ۲: ...
...
```

اگر گروه‌بندی فصل انجام نشود، همین جدول به‌صورت یک لیست تخت (بدون سرفصل فصل) رندر می‌شود.

**نکته‌ی زبان:** دو فایل کاملاً مستقل و کامل هستند (نه یک فایل دوزبانه) — یکی صد‌درصد فارسی، دیگری صد‌درصد انگلیسی، هر دو با همان اعداد صفحات و زمان (چون اینها زبان‌مستقل‌اند)، فقط عنوان/موضوع/برچسب‌ها ترجمه می‌شوند.

## ۶. رابط MCP

| Tool | ورودی | خروجی |
|---|---|---|
| `split_document` | `file_path`, `min_pages`, `max_pages` | لیست چانک‌ها + مسیر `manifest.json` |
| `get_chunk` | `chunk_id` | محتوای یک چانک مشخص |
| `verify_integrity` | `output_dir` | نتیجه‌ی verification |
| `generate_study_index` | `output_dir`, `reading_speed_wpm` | دو فایل `study-index-fa.md` و `study-index-en.md` |

رجیستر شدن در Claude Code از طریق:
```
claude mcp add doc-splitter -- node server.js
```
یا از طریق فایل `.mcp.json`. چون MCP استاندارد باز است، سایر ابزارهای پشتیبان (Cursor، Windsurf، Cline) هم بدون کد اضافه همین سرور را می‌شناسند.

### حالت مستقل (Standalone CLI)
همان Orchestrator، پشت یک CLI ساده:
```
doc-splitter --input book.pdf --out ./chunks --min-pages 5 --max-pages 10
```

## ۷. پارامترهای قابل‌تنظیم

| پارامتر | مقدار پیش‌فرض | توضیح |
|---|---|---|
| `min_pages` | 5 | حداقل اندازه‌ی هر چانک |
| `max_pages` | 10 | حداکثر (نرم، نه سخت — طبق قانون طلایی) |
| `boundary_window_pages` | 15 | بازه‌ی اولیه‌ای که در هر مرحله به‌صورت کامل به مدل داده می‌شود تا مرز مفهومی واقعی را تشخیص دهد (بزرگ‌تر از max_pages، به‌عنوان بافر دید) |
| `boundary_window_extension_pages` | 10 | مقدار افزایش بازه در هر بار که مدل تشخیص دهد مفهوم هنوز کامل نشده |
| `words_per_page` | 400 | برای تبدیل «صفحه» به «تعداد کلمه» در DOCX/متن بدون صفحه‌بندی فیزیکی |
| `image_extraction` | true | استخراج و ذخیره‌ی جداگانه‌ی تصاویر |
| `reading_speed_wpm` | 200 | سرعت مطالعه برای محاسبه‌ی زمان تقریبی هر بخش در ایندکس |
| `use_llm_chapter_grouping` | false | فعال‌سازی گروه‌بندی معنایی فصل‌ها با LLM، وقتی ساختار heading اصلی کافی نیست |
| `ocr_enabled` | false | همیشه غیرفعال در این نسخه؛ اسناد اسکن‌شده پشتیبانی نمی‌شوند |
| `on_missing_text_page` | `skip_and_flag` | رفتار هنگام برخورد با صفحه‌ی بدون متن قابل‌استخراج: آن صفحه رد می‌شود، در گزارش علامت‌گذاری می‌شود، و پردازش بقیه‌ی سند ادامه پیدا می‌کند |
